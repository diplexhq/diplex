# Руководство разработчика

## Принципы разработки

- **Ноль зависимостей** — используется только стандартная библиотека Go
- **Бережливое потребление ресурсов** — бенчмарки и оптимизация производительности
- **Самоприменимость** — DIplex генерирует DI-контейнер для самого себя
- **Тестовое покрытие** — все фичи реализованы в `internal/tests/`, там же основное тестовое покрытие
- **Минимализм** — не усложнять без необходимости
- **Производительность** — любые накладные расходы должны быть обоснованы
- **Типобезопасность** — проверять всё возможное на этапе компиляции
- **Читаемость** — сгенерированный код должен быть понятен

## CI / Git-хуки

Git pre-commit hook (`.githooks/pre-commit`) запускается на проиндексированных `.go` файлах:
1. `gofmt -w` — автоформатирование
2. `golangci-lint` — линтинг
3. `go vet` — статический анализ
4. `go build ./...` — проверка компиляции

Установка: `cp .githooks/pre-commit .git/hooks/pre-commit && chmod +x`

## Технологический стек

- Go 1.26+
- Только стандартная библиотека
- Парсинг AST через `go/parser`
- Шаблоны через `text/template`
- Конкурентность через `sync.WaitGroup` (4 горутины для парсинга, 1 для сканирования)

## Стиль кода

**Импорт алиасы:** имена пакетов в snake_case должны использовать camelCase-алиасы.

```go
// ПРАВИЛЬНО
import fileScanner "github.com/diplexhq/diplex/internal/file_scanner"
```

## Обработка ошибок

**Паника — единственный механизм обработки ошибок:**

- Сбои на этапе генерации = фатальны по своей природе
- Используйте `panic("message")` — ошибки флагов, отсутствие файлов, сбои парсинга
- Никогда не используйте `os.Exit()` напрямую
- Щедро используйте `utils.Must()` и `utils.NoErr()`

При обнаружении некорректного Go-кода всегда вызывайте панику с описательным сообщением:

```go
panic("corrupted Go source: empty identifier name — fix your source code")
```

## Обзор архитектуры

```
main.go
  └─ diplex.NewDI() → DI facade (config, logger, scanner, parser, resolver, generator)
       │
       ├─ Scanner.Scan() → domain.SourceFiles (channel)
       ├─ Parser.Parse(SourceFiles) → domain.ParsedData (providers + interfaces)
       ├─ Resolver.Resolve(ParsedData) → domain.ResolvedData
       └─ Generator.Generate(ResolvedData) → .go files
```

### Парсер
Анализирует Go исходные файлы и извлекает: провайдеры (`New*` функции), интерфейсы, реализации (методы структур с флагом `RealType: true`), тип-алиасы и embeds.

### Ресолвер
Строит индексы (`typeIndex`, `interfaceMethodIndex`) и разрешает все зависимости провайдеров для DI-фасадов через BFS обход.

### Генератор
Создаёт код DI контейнера из разрешённых данных через Go-шаблоны (`facade.tmpl`, `provider.tmpl`, `head.tmpl`, `import.tmpl`).

## Проверка сгенерированного кода по хешу

Файл `internal/tests/di_test.go` содержит интеграционный тест `TestDI`, который генерирует DI-контейнер и проверяет его SHA-256 хеш.

- Хеш хранится как константа `expectedHash`
- При изменении логики генерации тест упадёт с сообщением о несовпадении хеша
- Для обновления: пересобрать тест `go run .` → `sha256sum internal/tests/generated/diplex/di.go` → обновить константу `expectedHash`

Это гарантирует детерминизм генерации и предотвращает случайные изменения в коде генератора.

## Самопроверка

```bash
# Сборка DI для самого проекта (без дополнительных аргументов, использует internal/di и internal/generated)
go run .

# Сборка DI для тестового матрица (указать целевую директорию через -scan)
go run . -scan internal/tests -di internal/tests/di -out internal/tests/generated/diplex
```

Оба должны компилироваться успешно (`go build ./...`).
