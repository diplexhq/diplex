# Руководство пользователя

## Установка

```bash
go get -tool github.com/diplexhq/diplex@latest
go mod vendor
```

## Быстрый старт

```bash
go tool diplex
```

## Флаги запуска

| Флаг | Описание | По умолчанию |
|------|----------|-------------|
| `-scan` | Через запятую перечисленные директории для сканирования провайдеров, интерфейсов и реализаций | `"internal"` |
| `-out` | Выходная директория для сгенерированных файлов DI-контейнера | `"internal/generated/diplex"` |
| `-di` | Директории, содержащие DI-фасад интерфейсы (интерфейс `DI`, от которого зависит ваше приложение) | `"internal/di"` |
| `-module` | Явный путь Go-модуля (переопределяет автоматическое определение из `go.mod`) | _(авто из `go.mod`)_ |
| `-skip` | Регулярное выражение для файлов/директорий, которые нужно пропустить при сканировании | `"(internal\/generated\/diplex|tests|mocks?|_test\.go|_mock\.go)$"` |
| `-v` | Подробный вывод — печатает детали генерации в stderr | — |
| `-s` | Тихий режим — подавляет все выводы | — |

**Взаимоисключение:** флаги `-v` и `-s` не могут использоваться одновременно — при их совместном указании произойдёт паника.

### Примеры

```bash
# Сканировать конкретные директории
go tool diplex -scan internal/pkg,internal/service

# Пользовательская директория вывода
go tool diplex -out pkg/generated/di

# Пропустить сгенерированные директории
go tool diplex -skip "(generated|vendor|_test\.go)$"

# Переопределить путь модуля (нет go.mod)
go tool diplex -module example.com/my/app

# Подробный режим
go tool diplex -v

# Тихий режим
go tool diplex -s
```

## Порядок работы

DIplex выполняет работу в четыре фазы:

### 1. Сканирование

Рекурсивно сканирует папку internal, игнорирует моки и файлы тестов.

### 2. Распознавание

DIplex выявляет две категории сущностей:

**Провайдеры** — функции, совпадающие с паттерном `New*`, которые возвращают экспортируемый тип. Первый возвращаемый тип — производимый объект; опциональный второй возвращаемый тип — `error`. Поддерживаются generic-провайдеры.

```go
// Распознаётся как провайдер
func NewConfig() *Config
func NewDB(cfg *Config) (*DB, error)
func NewRepo[T User | Order]() *Repo[T]  // → NewRepo[User], NewRepo[Order]
```

**Интерфейсы** — как чистые декларации, так и конкретные реализации на типах. Поддерживается встраивание как для интерфейсов, так и для структур, поддерживаются дженерики для реализациях на типах.

```go
// Распознаётся как интерфейс
type UserStore interface {
    Get(id int) (*User, error)
}
```

```go
// Распознаётся как реализация интерфейса
type userRepository struct { db *DB }
func (r *userRepository) Get(id int) (*User, error) { ... }
```

### 3. Разрешение

Зависимости разрешаются автоматически. Ресолвер сопоставляет провайдеры с параметрами потребителей по типу и имени параметра.

- **Циклические зависимости** не обнаруживаются — при наличии цикла инициализация вызывает панику во время выполнения.
- **Слайс-зависимости** собирают все подходящие провайдеры автоматически.
- **Несколько реализаций** — если зависимость может быть удовлетворена несколькими провайдерами, неоднозначность разрешается сопоставлением имени результата провайдера с именем параметра потребителя. Если неоднозначность или отсутствие совпадения — паника.

```go
// Разрешение неоднозначности по имени параметра
func NewUserRepo(userRepo *UserRepo) { ... }
// ресолвер сопоставляет параметр `userRepo` с результатом `UserRepo`

func NewStorage(redisDsn Dsn) { ... }
// ресолвер сопоставляет параметр `redisDsn` с `NewRedisDsn`, возвращающим `(redisDsn Dsn)`
```

### 4. Генерация

Генерируется стабильный код в `-out`, создающий имплементации для интерфейсов фасадов.

## Результат

После успешного выполнения в `-out` появляется `.go` файлы

- Структуры с полями для каждой разрешённой зависимости
- Конструкторы `New*()`, создающие имплементации фасадов
- Ленивая инициализация на базе `sync.OnceValue` — зависимости создаются только при первом обращении

Один DI-фасад можно переиспользовать в разных сборках (веб-сервер, крон, CLI, обработчик очереди). 
Для удобства можно определять отдельные фасады на каждую сборку, где будут только нужные провайдеры.

### Использование сгенерированного контейнера

```go
// В вашем проекте — объявите фасад
// internal/di/di.go
package di

type DI interface {
    Config() *config.Config
    DB() *db.DB
    UserRepository() *userRepository
}
```

```go
// После запуска `go tool diplex`:

func main() {
    deps := diplex.NewDI()  // все зависимости уже спаяны

    cfg := deps.Config()
    db := deps.DB()
    repo := deps.UserRepository()

    // ... используйте сервисы
}
```

## Ограничения

### Сопоставление типов

- **Примитивные типы сопоставляются по имени** — `string`, `int`, `int64`, `bool` и другие встроенные примитивы сопоставляются по имени аргумента и имени/типа результата. Используйте именованный тип или алиас вместо чистого типа (например, `type Port int` или `type Port = int`), а не голый `int` или `string`.
- **Внешние интерфейсы не сканируются** — учитываются только интерфейсы, объявленные в исходных файлах проекта. Интерфейсы стандартной библиотеки и сторонних пакетов невидимы для ресолвера.

### Правила провайдеров

- **Подпись возвращаемых значений** — провайдеры должны возвращать ровно 1 или 2 значения. Если 2, второе должно быть `error`. Функции, возвращающие 3+ значений, молча игнорируются.
- **Только синглтоны** — все провайдеры инстанцируются один раз и кэшируются через `sync.OnceValue`. Семантики фабрики / вызова на каждую операцию нет.
- **Циклические зависимости** — вызывают panic при запуске из-за бесконечной рекурсии конструкторов. Ресолвер не обнаруживает циклы на этапе компиляции; это известное ограничение подхода BFS-разрешения.

### Правила интерфейсов

- **Полная поддержка узких интерфейсов** — поддерживаются узкие интерфейсы по месту использования, но для слишком узких необходимо обеспечить возможность разрешить неопределённость по имени аргумента (например, разные имена параметров для разных реализаций одного интерфейса).
- **Теги, веса, приоритеты через интерфейс** — в DIplex нет встроенных тегов, весов или приоритетов. Если вам нужны такие возможности, объявите интерфейс с методами `ID() string`, `Name() string`, `Weight() int`, `Priority() int` и т.п., реализуйте его на ваших типах и в провайдере запросите слайс (`[]MyInterface`). Ресолвер соберёт всех подходящих провайдеров в список, который вы можете отсортировать или отфильтровать во время выполнения.

### Правила generic типов

- **Обеспечьте возможность сузить до конкретного варианта** — комбинаторный список дженерик-провайдеров поддерживается, но должен сводиться к конечному количеству комбинаций. Это довольно странная вещь и для проработки нужны сценарии реального использования.

### Композитные типы

- **Сложные inline типы** — ресолвер сопоставляет `map[string]string`, `chan<- Event`, `<-chan Result` и другие composite типы в сигнатурах методов. Сложные inline типы с функциями, интерфейсами или структурами тоже сопоставляются. Однако рекомендуется избегать такой сложности — это плохой стиль, редко используется и поэтому не покрыто тестами. Именованные типы сопоставляются автоматически; type aliases не меняют работу ресолвера.
